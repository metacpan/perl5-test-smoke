The MetaCPAN Design System is a complete, production-ready foundation for building consistent, accessible, and unmistakably crafted experiences across the Comprehensive Perl Archive Network. Editorial in voice, monospaced in soul, and rigorously tokenized for the engineers who depend on it.

01 — Foundations

Core principles

Seven commitments that guide every design decision. They protect the legacy, serve the developer, and keep the system honest.

P-01

Legible above all

Module names, signatures, and error messages must be unambiguous. We optimize for the developer reading at 3 a.m., not the marketing screenshot.

P-02

Honor the heritage

Perl is older than most engineers using it. The system carries that weight with editorial typography and a mature, restrained palette — no toy aesthetics.

P-03

Density with air

Developers scan dense pages of metadata. We use tight vertical rhythm where data lives and generous breathing room where decisions happen.

P-04

Accessible by default

Every token, every component clears WCAG 2.2 AA at minimum. Focus is visible, motion is respectful, and the keyboard is a first-class citizen.

P-05

One brand, two modes

Light and dark are equal expressions of the brand — not afterthoughts. Tokens flip; intent never does.

P-06

Tokens, not exceptions

If you reach for a hex code or a magic pixel value, you've found a gap in the system. Fix the system, not the screen.

P-07

Build for the long run

CPAN has shipped since 1995. Choose patterns that age — not trends that expire. Boring, well-built, and predictable wins.

Every foreground/background pairing in the system is verified against WCAG 2.2. No combination falls below AA. Body text and primary actions reach AAA wherever feasible.

Usage guidelines

Do — Color
  • Use --accent for the single most important action per view.
  • Pair semantic tokens with iconography — color alone never conveys meaning.
  • Use teal/500 for links and security-related communications.
  • Reserve red/700+ for the dark sidebar surface or dense data tables.
  • Test every new color pair against contrast tokens before shipping.
Don't — Color
  • Don't use --accent for body text or large fills.
  • Don't introduce new hex values — extend the system instead.
  • Don't mix red and danger semantically; danger is its own token family.
  • Don't tint dark mode by inverting light mode — use the dark theme tokens.
  • Don't rely on red/green pairings without textual or iconographic backup.

Responsive behavior

Display sizes use clamp() to scale fluidly between viewport breakpoints. Body sizes are fixed — readability is non-negotiable. The opsz axis on Fraunces is bound to the literal pixel size, so optical compensation happens automatically.

/* Fluid display, fixed body */
.hero-title {
  font-size: clamp(3rem, 7vw, 4.75rem);
  font-variation-settings: 'opsz' 144;
}

.body { font-size: 1rem; /* never scales */ }

Component anatomy — primary button (canonical example)

Every component in the library is documented to this depth. Below is the canonical reference: anatomy, states, accessibility contract, and code spec.

Anatomy
  • Container — bg --accent, radius --radius-md, height 40px
  • Label — Inter 500, --text-sm, --fg-on-brand
  • Icon (optional) — 16×16, currentColor, gap --space-2
  • Padding — --space-4 horizontal
  • Focus ring — --shadow-focus on :focus-visible
Accessibility
  • Native <button> element required (or role="button" + key handlers)
  • Min target 40×40px (44 on touch)
  • Contrast ≥ 5.74:1 (white on red/500) — WCAG AAA for large text
  • Disabled state uses aria-disabled, not just opacity
  • Loading state uses aria-busy="true" with visible spinner
React · button.tsxcanonical
import { forwardRef } from 'react';
import { cva, type VariantProps } from 'class-variance-authority';

const button = cva(
  'inline-flex items-center justify-center gap-2 rounded-md font-medium transition focus-visible:outline-none focus-visible:ring-3 focus-visible:ring-accent/30 disabled:opacity-45 disabled:pointer-events-none',
  {
    variants: {
      variant: {
        primary:   'bg-accent text-on-brand hover:bg-accent-hover active:bg-accent-press',
        secondary: 'bg-surface text-fg border border-default hover:border-strong',
        ghost:     'bg-transparent text-fg hover:bg-subtle',
        danger:    'bg-danger-500 text-white hover:bg-danger-700',
      },
      size: { sm: 'h-8 px-3 text-xs', md: 'h-10 px-4 text-sm', lg: 'h-12 px-5 text-base' },
    },
    defaultVariants: { variant: 'primary', size: 'md' },
  }
);

export const Button = forwardRef<HTMLButtonElement, ButtonProps>(
  ({ variant, size, loading, children, ...props }, ref) => (
    <button ref={ref} className={button({ variant, size })}
      aria-busy={loading} disabled={props.disabled || loading} {...props}>
      {loading && <Spinner />}
      {children}
    </button>
  )
);
09 — Guidelines

Design patterns

Repeated solutions to recurring product problems. Each pattern is composed entirely of primitive components and follows the principles set in section 01.

PATTERN · 01

Search-first navigation

The search input is the dominant element on every primary surface — never demoted. ⌘K opens it from anywhere. Filter chips appear below the input, never above.

PATTERN · 02

Module result row

Module name (mono) → tagline (sans) → metadata strip (mono, tertiary). The eye lands on the namespace first, every time.

PATTERN · 03

Code-first documentation

Every doc page leads with an install command and a 5-line synopsis. Prose comes after. The developer can leave with what they need in 10 seconds.

PATTERN · 04

Form layout

Single column. Labels above fields. Helper text below. Errors inline with red border + icon + message — never red text alone.

PATTERN · 05

Empty state recovery

Every empty state proposes a next action: a broader query, a featured module, or a link to the docs. We never strand the user.

PATTERN · 06

Destructive confirmation

Two-step confirmation for permanent actions. The destructive button is always on the right; cancel is always neutral and on the left.

10 — Guidelines

Do's & don'ts

A consolidated reference covering the most common decision points. When in doubt, return here.

Do — Typography
  • Use Fraunces only for headings, hero copy, and statistical figures.
  • Use JetBrains Mono for module names, code, and CLI commands — always.
  • Limit body line-length to ~64 characters for readability.
  • Apply opsz on Fraunces matching the rendered px size.
Don't — Typography
  • Don't use Fraunces for body — it's display-only.
  • Don't render module names in proportional fonts — characters matter.
  • Don't use uppercase for body or paragraph text.
  • Don't introduce new font families.
Do — Components
  • Compose pages from existing components — extend before you replace.
  • Provide focus-visible states for every interactive element.
  • Use semantic colors via tokens, not raw hue values.
  • Test every component in dark mode before shipping.
Don't — Components
  • Don't fork components — file an RFC if the system is missing something.
  • Don't suppress focus rings, even on hover.
  • Don't combine more than two elevation levels in one view.
  • Don't use motion durations longer than 500ms for UI transitions.
Do — Layout
  • Stick to the 12-column grid for page-level layout.
  • Use the 8pt scale for all spacing values.
  • Maintain consistent vertical rhythm across sections.
  • Honor the 1280px max content width on wide displays.
Don't — Layout
  • Don't use arbitrary pixel values like 7px or 22px.
  • Don't break out of the grid for "design moments" — use space, not chaos.
  • Don't full-bleed text content past 80 characters wide.
  • Don't nest containers with the same elevation.

12 — Guidelines

Developer handoff

Everything an engineering team needs to ship MetaCPAN UI in production. Steps are ordered; skipping ahead breaks downstream guarantees.

STEP · 01

Install tokens

Pull @metacpan/tokens via npm. Run Style Dictionary at build time to generate platform-specific outputs (CSS variables, JS module, iOS plist, Android XML).

$ pnpm add @metacpan/tokens
$ pnpm tokens:build
STEP · 02

Wire CSS variables

Import the generated tokens.css at the application root. Toggle dark mode via [data-theme="dark"] on <html> — no class swaps, no JS needed for the theme to apply.

STEP · 03

Use the React kit

Components ship as @metacpan/ui. Each is fully typed, server-render-safe, and uses CVA for variant management. Tree-shakable per-component imports.

import { Button, SearchInput } from '@metacpan/ui';
STEP · 04

Verify accessibility

Run axe-core in CI for every PR touching UI. Manual keyboard pass is required for any new pattern. Screen-reader pass on VoiceOver + NVDA before promotion to main.

STEP · 05

Document deviations

If a screen requires a token or component the system doesn't yet provide, file an RFC in @metacpan/ds-rfcs. Bypassing the system is a P1 design debt.

STEP · 06

Versioning

SemVer strict. Token renames or color value changes are major. Component prop additions are minor. Bug fixes are patch. Migration codemods are shipped with every major.
